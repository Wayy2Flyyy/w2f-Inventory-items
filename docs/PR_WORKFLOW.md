# Pull Request Workflow

## Standard workflow

1. Create a branch from the current main branch.
2. Make focused changes for one objective.
3. Use meaningful commit messages.
4. Open a pull request with clear summary and testing notes.
5. Review the files changed tab before requesting merge.
6. Run validation tooling.
7. Merge only when checks and review are clean.
8. Avoid unrelated changes in the same PR.
9. Update the changelog when user-facing behavior changes.

## Good commit message examples

- `docs: add installation guide`
- `items: add mechanic item definitions`
- `tools: improve validation report`
- `ci: add validation workflow`
- `chore: update changelog`

## Bad commit message examples

- `update`
- `fix`
- `stuff`
- `test`
- `commit`
- `more changes`
